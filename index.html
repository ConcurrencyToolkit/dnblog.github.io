Text IO Performance

October 19, 2022 · 5 min read

In the previous post, I discussed the usability flaws of plain-text input parsing in .NET platform. In this post, we will focus on performance issues related to plain-text input and output.

Plain-text IO performance is a big issue for competitive programming. If the input or output is too large, the solution may get Time Limit or Memory Limit exceeded verdict due to inefficient IO. IO will be a bottleneck.

Consider the problem Timus: 1510. Even a tricky C# solution with optimal O(n)O(n)O(n) complexity got Memory Limit Exceeded verdict due to inefficient input parsing. But nonoptimal C++ solution with O(nlog(n))O(nlog(n))O(nlog(n)) complexity easily got Accepted. The choice of language is a game changer in this case.

October 6, 2022 · 5 min read

In competitive programming contests, interviews, or when solving problems from archives for learning we are facing automatic judging systems. These systems accept code on some supported language, then compile and run it against many test cases.

Although modern versions of C# are usually supported on these systems, I prefer to use C++, Java, or Kotlin for solving algorithmic problems. Why? Because judging systems provide test cases as plain text to STDIN and C# doesn't have a good way to read and parse it.
